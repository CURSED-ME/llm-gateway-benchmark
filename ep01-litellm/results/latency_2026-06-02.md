# Proxy Overhead Latency Test Results (Loopers vs. LiteLLM)

**Date:** 2026-06-02
**Test Goal:** Measure the latency the proxy itself adds, isolated from upstream time.

## Setup
- **Concurrency:** 500 VUs for 60 seconds
- **Mock Upstream Delay:** 50ms artificial delay

## Results: What does your proxy add to every single LLM request, forever?

LLM calls are already slow (500ms–30s). A well-implemented proxy should be invisible. We measure P99 latency overhead because the 99th percentile represents real pain for real users.

*Note: Proxy overhead is calculated as (Measured Latency - 50ms upstream delay).*

| Metric | Loopers Overhead | LiteLLM Overhead |
|---|---|---|
| P50 Latency | 52.63 ms | 1,445.41 ms |
| P90 Latency | 140.89 ms | 2,344.56 ms |
| P99 Latency | 240.98 ms | 46,812.60 ms |

### Error Rates & Stability
During this 500-VU test:
- **Loopers** maintained stability, successfully processing 260,793 requests with a 0.95% failure rate.
- **LiteLLM** buckled under the concurrency, dropping 22.53% of its 13,412 requests due to timeouts and socket hangs. This explains the extreme P99 latency overhead (46.8 seconds), as requests piled up in the Python asyncio event loop before eventually timing out.

### Notes on Acceptance Criteria
The expected threshold for Loopers was `< 2ms` P99 overhead. Because this test was run under heavy load (500 concurrent VUs), the proxy overhead includes significant queueing time, pushing Loopers' P99 overhead to 240.98ms. While Loopers did not hit the <2ms goal under these saturated conditions, it remained fundamentally stable compared to LiteLLM's catastrophic queuing failure. As dictated by our transparency rules, we publish these trade-offs honestly.
